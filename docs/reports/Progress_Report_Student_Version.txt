1. Introduction - Why I'm Doing This Project

Sri Lanka is developing really fast, but the way we plan our cities hasn't really caught up. While traveling around the country, I’ve seen so many new buildings popping up in places that are obviously not suitable—either they are on sensitive land or in areas that flood every single year. It’s frustrating because it feels like we are creating problems for the future. This is especially true in Colombo North, like in Modara and Grandpass. These areas are super crowded, and they are right next to the Kelani River, which makes them a "perfect storm" for flooding.

Right now, if you want to see if a piece of land is safe to build on, you have to find old PDF maps from the UDA or look through books of NBRO safety rules. It’s a very manual process and honestly, it’s confusing for most people. I decided to build a "GeoAI Agent" to fix this. Instead of a person digging through maps, they can just ask a question in plain English, and the system does all the heavy lifting—scanning the maps and the rules to give a solid recommendation.

2. Expanding the Scope and Gathering Messy Data

At first, I was only looking at a few wards in Colombo North. But I realized pretty quickly that floodwaters don’t care about administrative boundaries. People living in Kelaniya or Angoda face the same river risks as those in Modara. So, I expanded my study area to include these expansion zones because they are all part of the same hydrological system. 

Finding good data was one of the hardest parts of this project so far. A lot of the official flood maps were just static images or PDFs, which the computer can’t really "read." I had to spend many hours in QGIS georeferencing these images and digitizing them into GeoJSON files so they could actually be used for spatial analysis. I've included a map in my findings called "Colombo North: Urban Infrastructure and Flood Risk Overlay" which shows exactly how these flood zones sit right on top of our dense housing areas. Seeing it visually like that really hits home how dangerous some of these locations are.

[INSERT IMAGE 2 HERE: MAP OF STUDY AREA OVERLAY - Showing the building footprints and road network with the red flood risk polygons]

I also used OpenStreetMap data for the basics like roads and buildings. But OSM isn’t always up to date, so I started using PlanetScope satellite imagery. These images are super clear (3-meter resolution) and give me a way to see new constructions that aren't on the official maps yet. It’s been a massive task just getting all this multimodal data into a state where it’s actually "AI-ready."

3. Spatial Data Engineering: The "Under the Hood" Stuff

This was the most technical part of my work. Geospatial data is incredibly messy. I had files in all sorts of different "coordinate systems" (like Kandawala or WGS84), and if they don’t match perfectly, the whole analysis is useless. I built a Python pipeline using GeoPandas to make sure every single piece of data is standardized. 

Setting up the PostGIS database was a big win. Standard databases aren't great with maps, but PostGIS lets me run spatial queries like "ST_Intersects." This means my system can instantly tell if a building is sitting in a high-risk flood zone. To see where the most pressure is, I generated a "Building Density Hotspots Heatmap." When I ran this, it was a bit of an "aha" moment—the heatmap shows these bright red clusters in the core developed areas, which are exactly the places most vulnerable to the river. It’s clear that we have too many people packed into the most dangerous zones.

[INSERT IMAGE 3 HERE: BUILDING DENSITY HEATMAP - The hexbin plot showing the red hotspots of urban concentration]

I also had to "simplify" the building data. With over 150,000 buildings, the rendering was getting really slow. By simplifying the shapes slightly, I managed to cut the data size by almost half while keeping the maps looking sharp and accurate.

4. Data Findings: What the Charts are Telling Us

One of the most interesting parts of my recent work was analyzing the "Public Infrastructure" apart from just houses. I created a "Distribution of Key Public Infrastructure Donut Chart" to see what else is at risk besides homes. I decided to exclude residential buildings from this specific chart because there are so many of them (over 150k!) that they just dwarfed everything else and made the chart impossible to read.

[INSERT IMAGE 4 HERE: INFRASTRUCTURE PIE/DONUT CHART - Showing the breakdown of Schools, Religious Places, etc.]

Once I cleared the "noise" of the houses, I could see that Religious Places and Schools make up the biggest chunks of our public infrastructure in this area. This is a huge concern because these are "critical facilities." If a major flood hits, these are the places people usually go to for shelter, but my maps show that many of these are located right in the high-risk zones. This kind of insight is exactly what I want my GeoAI agent to be able to explain to a user.

5. Progress on the Dashboard and the Agent Core

To make sure all this data was actually useful, I built a visualization dashboard. I had to migrate it from Leaflet to MapLibre GL JS because Leaflet just couldn't handle the 150,000 buildings. Now, it uses the computer's graphics hardware, and panning around Colombo North is finally smooth. I used a "glassmorphism" look for the UI to make it feel like a modern tool that a professional planner would actually want to use.

[INSERT IMAGE 5 HERE: DASHBOARD UI SCREENSHOT - A mockup or real screenshot of the high-performance map interface]

But the biggest shift is what I'm doing now: building the "brain" or the agentic part. I’m using LangChain and a framework called ReAct. The idea is that the agent doesn't just guess an answer; it uses "tools" I’ve built—like one to check flood risk and another to look up land-use rules. It follows a "Chain of Thought" process. If you ask, "Can I build a clinic here?", the agent thinks: "First, I need to check the flood risk... then I need to check if the area is zoned for healthcare... then I'll give my advice."

I’m also setting up a RAG (Retrieval-Augmented Generation) pipeline. This is so the agent can "read" the actual UDA city planning documents. I’ve started indexing the rules into a vector database (ChromaDB) so the agent can cite specific sections of the law. This is the best way to stop the AI from "hallucinations" or making up rules.

6. Challenges and Next Steps

The next big step is finishing the RAG integration and then testing how smart the agent really is. I’m planning to compare its recommendations against traditional planning models (like MCDA) to see if it’s as accurate as a human urban planner. 

It’s been a really rewarding but challenging project so far. Dealing with messy data was a headache, and getting the AI to "reason" through spatial problems is tough, but seeing the maps and charts come together makes it feel like it's finally working. This system has the potential to really help keep people safe by making urban planning much smarter.
